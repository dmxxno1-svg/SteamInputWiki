本章并非贬低原生 SIAPI 或混合输入（Mixed Input）；理想情况下开发者两者兼顾，但若无法两者兼顾，应选择最适合游戏的输入方案。

若要让游戏在 Steam 硬件上有良好体验，建议支持 Mixed Input 或 Native Steam Input（视项目情况而定）。两者各有优劣，错误的实现都会带来问题。

## 评估应采纳哪种实现的要点

Steam Input 是一套功能强大的现代输入 API，支持多数当前控制器、对触觉的低层控制、以及自动切换按键提示等功能。但并非适合所有项目，主要有以下几类情况可能不适合：

- A) 小团队/预算有限：可能没有资源实现高质量的 SIAPI，独立游戏可优先考虑优化对混合输入的支持并添加若干贴心功能（如硬件提示、官方配置）
- B) 游戏设计较为简单：如果游戏只需少量按键（如平台跳跃类），直接实现完整 SIAPI 可能得不偿失，可选择部分支持（例如仅自动切换提示图标）
- C) 非 Steam 平台或多平台发行：SIAPI 依赖 Steamworks SDK，若不以 Steam 为主要发布平台，团队可能不愿为此投入额外维护成本

若你属于上述情形之一，可优先保证传统控制器与混合输入的良好支持，而非强行实现 SIAPI。

## 当不使用 SIAPI 时的替代选项

若决定不优先实现 SIAPI，请考虑其它输入方案：

* DirectInput：兼容性旧但可用于向后兼容，不建议仅用作现代项目的主输入 API
* XInput：易于移植但功能有限，仅针对 Xbox 风格控制器，对非 Xbox 控制器支持不足
* Unity/Unreal 引擎输入系统：集成方便，适合快速支持常见控制器，但对陀螺等高级特性需额外插件
* SDL2：设备兼容性好，是较为前瞻的选择，但与 SIAPI 在陀螺原生支持上并不完全等效

## 在游戏中实现混合输入（Mixed Input）的建议

玩家期望的是混合输入不会导致问题（如卡顿、输入被限制为摇杆值等）。一些关键建议：

- 不把鼠标移动本身直接转换成图标切换的判据，推荐用鼠标点击事件来切换回鼠标/键盘提示
- 将瞄准辅助与手柄使用解耦，并提供开关与强度调节选项
- 在匹配策略上，现阶段建议把混合输入玩家视作鼠标键盘用户来匹配；未来陀螺用户增多时可考虑独立分组
- 提供仅图标类型的手动切换选项（例如强制显示 Steam 硬件提示）以兼容特殊场景

总体而言，支持混合输入通常并不复杂，制作并附带一个或数个官方控制配置往往能显著改善用户体验。
